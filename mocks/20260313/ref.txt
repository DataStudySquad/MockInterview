設計一個像 KKTIX 這樣的搶票系統，最大的挑戰在於**極高併發（High Concurrency）**的瞬間流量。當熱門演唱會開賣時，系統必須在幾秒鐘內處理數十萬甚至上百萬次的請求，同時確保**不超賣**且**資料一致**。

以下是針對「高性能」目標設計的系統架構方案：

---

## 1. 核心設計架構圖概念

系統應採用**微服務架構**與**非同步處理**，將流量壓力分層消化。

### 核心分層：

* **流量緩衝層 (Traffic Layer):** 透過 CDN、負載平衡器（LB）與 API Gateway 阻擋無效請求與機器人。
* **排隊層 (Queuing Layer):** 利用 Redis 或 Kafka 建立虛擬等候室，避免資料庫直接崩潰。
* **訂單執行層 (Order Processing):** 快速扣減庫存，異步生成訂單。

---

## 2. 關鍵高性能策略

### ### A. 動靜態分離 (Static/Dynamic Separation)

* **靜態資源：** 活動頁面的圖片、CSS、JS 等全部放在 **CDN** (如 Cloudflare, CloudFront)。
* **頁面靜態化：** 開賣前將活動資訊生成靜態 JSON/HTML。使用者進場時不需要查詢資料庫，直接從 CDN 獲取活動細節。

### ### B. 虛擬等候室 (Virtual Waiting Room)

不要讓所有流量直接衝向資料庫。

* **排隊機制：** 當流量超過預設門檻，將請求導向一個排隊頁面。
* **令牌桶 (Token Bucket)：** 只有持有「令牌」的使用者才能進入結帳流程，以此嚴格控制後端服務的承載量。

### ### C. 高性能庫存扣減 (Inventory Management)

**絕對不能直接操作 SQL Database 進行 `UPDATE stocks - 1**`，這會產生資料庫鎖定（Lock）瓶頸。

* **Redis 預扣庫存：** 將票券庫存存放在 Redis 中（使用 Lua Script 確保原子性）。
* `DECR stock_count`：若回傳值 $\ge 0$ 則代表搶票成功。


* **異步寫入：** 搶票成功後，發送訊息到 **Kafka/RabbitMQ**，後端訂單服務再慢慢從隊列中讀取訊息並寫入資料庫。

---

## 3. 系統組件詳細說明

| 組件名稱 | 技術選型建議 | 作用說明 |
| --- | --- | --- |
| **負載平衡** | Nginx / AWS ALB | 分散流量，進行初步的流量清洗。 |
| **分散式快取** | Redis | 存放庫存（Atomic Counter）與 Session，提供極速讀寫。 |
| **訊息隊列** | Kafka | 削峰填谷（Traffic Shaving），將訂單建立異步化。 |
| **主資料庫** | PostgreSQL / MySQL | 存放最終訂單與使用者資料，使用讀寫分離。 |
| **分散式鎖** | Redlock / ZooKeeper | 確保在極端情況下（如多機房同步）不會發生超賣。 |

---

## 4. 防止機器人與惡意刷票

高性能不只是快，還要「有效」。

* **驗證碼機制 (CAPTCHA):** 阻擋腳本程序，建議使用隱形驗證碼（如 Google reCAPTCHA v3）。
* **頻率限制 (Rate Limiting):** 針對單一 IP 或單一 User ID 設定每秒請求上限。
* **黑名單系統:** 自動識別並封鎖異常高頻的設備。

---

## 5. 資料一致性保障 (Reliability)

雖然使用 Redis 預扣庫存很快，但必須解決「使用者搶到票但不付款」的問題：

1. **預扣（Locking）：** Redis 扣減庫存後，給予 10 分鐘付款時限。
2. **釋放（Release）：** 若逾時未付款，透過監聽過期事件或定時任務，將庫存加回 Redis 並釋出。
